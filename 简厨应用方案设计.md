## 一、项目背景与可行性分析

### 1.1 行业与市场分析

根据最新统计数据，中国手机用户规模已达 14.2 亿，智能手机几乎实现全民覆盖。移动互联网深度渗透到日常生活的方方面面，其中「美食烹饪」类应用长期占据生活服务类下载榜前列。

在日常场景中，普遍存在以下痛点：

- **冰箱里有食材，不知道做什么**：用户拥有原材料但缺乏烹饪灵感。
- **想学做某道菜，不知道具体步骤**：有目标菜品但缺乏详细做法指导。
- **菜谱应用信息过载**：主流应用内容庞杂、广告繁多，新手用户难以快速找到简单实用的菜谱。

「简厨」正是针对以上痛点，定位为一款**轻量、简洁、专注**的菜谱工具应用。

### 1.2 竞争对手分析

| 产品 | 优势 | 劣势 | 简厨的差异化 |
|------|------|------|------------|
| **下厨房** | 菜谱丰富多样、美食社区活跃、时令推荐、餐具食材推荐 | 无法按食材反向搜索、广告穿插影响体验 | 按食材/菜名双维度搜索，无广告干扰 |
| **中华菜谱大全** | 分类详细、多数菜谱附视频教程、支持用户上传 | 同一菜品做法参差不齐，用户上传质量难保证 | 官方精选菜谱，做法统一可靠 |

### 1.3 自身条件分析

**核心优势：**

- **界面简洁美观**：遵循 Material Design 规范，视觉体验统一。
- **操作门槛低**：两种核心用法——按菜名搜索、按食材搜索，覆盖主要使用场景。
- **定时提醒功能**：支持早餐/午餐/晚餐三餐定时推送提醒，融入用户日常生活节奏。
- **社交分享**：一键分享菜谱至社交平台，口碑传播便捷。

## 二、产品定位与目标

### 2.1 产品定位

简厨是一款**轻量级菜谱工具应用**，致力于在用户烹饪遇到困惑时，提供快速、精准的菜谱查询与烹饪指导。

### 2.2 目标用户群

| 用户类型 | 场景描述 | 对应功能 |
|---------|---------|---------|
| **想学做菜的新手** | 想烹制某道菜品，但不清楚所需材料和步骤 | 按菜名搜索 → 查看详情页的原料列表和步骤指南 |
| **有食材但不知道做什么的用户** | 冰箱里有若干食材，不知道能组合出什么菜 | 按食材标签搜索 → 浏览匹配菜谱 |
| **需要饮食规划的用户** | 希望按时就餐，需要外部提醒 | 偏好设置页开启定时提醒，系统在设定时间推送通知 |

### 2.3 产品目标

1. **短期目标**：完成核心功能开发（菜谱浏览、详情查看、定时提醒），通过课程验收。
2. **中期目标**：完善食材搜索功能，扩充菜谱数据库至 200+ 条，优化 UI 细节。
3. **长期目标**：引入用户收藏、菜谱评分、社区分享等社交功能，扩大用户规模。

### 3.2 核心数据模型

```kotlin
data class Recipe(
    val id: Int,              // 唯一标识
    val title: String,        // 菜品名称
    val cookingTime: String,  // 烹饪时间（如 "3 Mins"）
    val prepTime: String,     // 准备时间（如 "1 Min"）
    val cover: String,       // 封面图片 URL
    val category: String?,    // 分类（breakfast/lunch/dinner）
    val ingredients: String?, // 原料清单（\n 分隔）
    val stepsGuide: String?   // 烹饪步骤（\n 分隔）
)
```

```
┌─────────────────────────────────────────────────┐
│                   View 层                        │

                     │ observe LiveData
                     ▼
┌─────────────────────────────────────────────────┐

│   MainViewModel（持有 UI 数据、通过协程发起请求）  │

└────────────────────┬────────────────────────────┘
                     │ 调用
                     ▼
┌─────────────────────────────────────────────────┐
│                Repository 层                      │
│   MainRepository（封装数据获取逻辑）              │
└────────────────────┬────────────────────────────┘
                     │ 调用
                     ▼
┌─────────────────────────────────────────────────┐
│                Network 层                        │
│   RetrofitBuilder（Retrofit 单例，Gson 转换）     │

                     │ HTTP GET
                     ▼
```

### 4.3 网络数据获取

- **API 地址**：`https://my-json-server.typicode.com/android-app-development-course/2020-A9-jianChu/db`
- **数据格式**：JSON，顶层包含 `recipes` 数组。
- **转换方式**：Retrofit + Gson 自动将 JSON 映射为 `RecipeData` / `Recipe` 数据类。
- **加载状态管理**：通过 `State<T>` 密封类封装 Loading → Success / Error 三态，UI 层根据状态显示加载动画、列表数据或错误提示。

### 4.4 定时提醒实现

```
用户开启开关（PreferencesFragment）
    │
    ▼
SharedPreferences 持久化存储开关状态
    │
    ▼
AlarmHandler.setAlarm()
    │
    ▼
AlarmManager.setAlarmClock() ← 设置精确闹钟
    │
    ▼  （到达设定时间）
AlarmReceiver.onReceive()
    │
    ▼
NotificationManager.sendNotification() ← 发送系统通知
    │
    ▼
用户点击通知 → 打开 MainActivity
```

### 4.5 多食材搜索方案

采用 **LIKE 语句多条件贪婪匹配算法**：

1. 对每个食材标签，使用 SQL `LIKE` 从数据库中筛选包含该食材的记录，仅选取主键。
2. 将各条件的筛选结果合并，按主键分组并统计匹配数量。
3. 按匹配数量降序排列——匹配度越高的菜谱排在越前面。
4. 根据排序后的主键回表查询完整记录并返回。

此算法的优势在于无需复杂的全文搜索引擎，在轻量级数据库中即可高效运行，适合菜谱数据量有限的场景。
